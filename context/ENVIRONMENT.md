## Pipeline runbook (one person, end to end)

Prereqs: gcsfuse mount active (quirk #11), reference at `~/ref/`, both pixi envs built.

```bash
# 0. Mount the bucket if not already (quirk #11)
mkdir -p ~/mnt/aou-controlled
gcsfuse --billing-project wb-glacial-potato-8710 --implicit-dirs vwb-aou-datasets-controlled ~/mnt/aou-controlled

# 1. Slice both technologies to the chr6 HLA region + convert to FASTQ (samtools lives in the spechla env)
cd ~/repos/pilot-validation && pixi shell -e spechla
bash scripts/slice_and_fastq.sh <id> <cram_path_rel_to_mount> <lr_bam_path_rel_to_mount>
#   paths e.g.:  pooled/wgs/cram/v8_base/wgs_<id>.cram
#                pooled/longreads/v8_delta/BI/revio/bam/<id>/GRCh38/<id>.bam

# 2. SpecHLA (short-read) — from ~/tools/SpecHLA, spechla env
{ time bash script/whole/SpecHLA.sh -n <id> \
    -1 ~/pipeline_outputs/<id>/R1.fastq.gz -2 ~/pipeline_outputs/<id>/R2.fastq.gz \
    -o ~/pipeline_outputs/<id>/spechla_output/ ; } 2> ~/pipeline_outputs/<id>/spechla_timing.txt

# 3. SpecImmune (long-read) — from ~/tools/SpecImmune, specimmune env
#    bwa config (proven; see DECISIONS re: minimap2) + visualization disabled (cosmetic, missing converter)
{ time python3 scripts/main.py -n <id> -o ~/pipeline_outputs/<id>/specimmune_output/ \
    -j 4 -y pacbio-hifi -i HLA -r ~/pipeline_outputs/<id>/LR.fastq --db ./db --visualization "" ; } \
    2> ~/pipeline_outputs/<id>/specimmune_timing.txt

# 4. 3-way comparison table (writes ~/pipeline_outputs/<id>/comparison.md; paste that back to fold into .local.md)
python3 ~/repos/pilot-validation/scripts/compare_hla_results.py <id>
```

Region window: **`chr6:29,500,000-33,500,000`** (4 Mb — classical MHC + ~440 kb pad each side). See DECISIONS / EXPERIMENTS for the open padding-narrowing experiment.

> Note: `slice_and_fastq.sh`'s *printed* SpecImmune hint omits `--visualization ""`; use the runbook form above — disabling it is the standing decision (DECISIONS). All project scripts referenced throughout this file live in `scripts/`; only upstream tools' own internal paths (`~/tools/SpecHLA/script/...`, `~/tools/SpecImmune/scripts/...`) are unaffected by this repo's layout.

## Workbench quirks & gotchas — read before doing anything new

1. **`pixi shell` only works from the manifest's directory.** `cd ~/repos/pilot-validation` first, always. Elsewhere it fails silently into a *plain, non-activated* shell (no error) and every tool command then breaks with confusing "command not found". Check for the `(omni-hla-pilot:<env>)` prompt prefix before trusting the shell.
2. **`pixi shell` sometimes misses its 3-second hook window.** It prints a fallback line like `. /tmp/pixi_env_XXX.sh` — run that line manually to finish activation. **Corollary:** any commands pasted in the *same block* as the `pixi shell` line get swallowed during this hiccup — paste run commands *separately*, only after the `(omni-hla-pilot:<env>)` prompt appears.
3. **Workbench's "Add repository" auto-clone never worked** this project — registering the repo, restarting the app, making it public: none cloned it into `~/repos/`. Don't debug it; just `git clone` manually.
4. **`Failed to get AoU version: no AoU resources in workspace`** prints on every fresh shell — benign noise, unrelated to data access (a real BigQuery query against the CDR returns real rows despite it).
5. **The CDR is not exposed via shell env vars** (`env | grep -i cdr` → nothing). It lives in a separate GCP project (`wb-silky-artichoke-2408`, dataset `C2025Q4R6`) and is accessed via BigQuery with a fully-qualified table path — the Cohort/Dataset Builder auto-generates this in exported notebook snippets (the intended access pattern).
6. **No sudo** (`jupyter` user not in sudoers) — can't apt-install anything. Work around missing system tools rather than installing (see the `less` fix).
7. **samtools in the `spechla` env (v1.21) has built-in GCS support** (`gs+http`/`gs+https`/`gs` handlers) — but it does **not** cleanly handle this requester-pays bucket (quirk #11); use the gcsfuse mount instead.
8. **Two collaborators, siloed apps, no conflict.** Aleix (`aleixruf@researchallofus.org`) works in his own app instance in the same workspace. Check his `hla-pilot-cohort-snapshot` resource before redoing the stratified ~100-person selection — it may already exist.
9. **Always confirm you're reading the current CDR release, not an archived one.** AoU republishes core docs per CDR version and keeps old ones live, marked "(Archived …)". Confirmed 2026-07-08: **v9 (`C2025Q4R6`) is current.** Re-check the "Current CDRs" table before trusting any cited fact from an older session.
10. **The platform migrated** from the original AoU Researcher Workbench (Terra/Firecloud) to **Verily Workbench** — visible in the `vwb-` vs legacy `fc-` bucket split and UI differences. Any doc/tutorial/memory using old-platform terminology needs live re-verification; when something doesn't match the screen, suspect staleness first, not user error.
11. **Don't use raw `gs://` URLs with samtools/htslib against the genomics bucket — use a `gcsfuse` mount.** htslib doesn't reliably attach billing to requests against this requester-pays bucket (`GCS_OAUTH_TOKEN` alone → `Invalid argument`; adding `GCS_REQUESTER_PAYS_PROJECT` → `Operation not permitted`; neither works). The confirmed working method:
    ```bash
    mkdir -p ~/mnt/aou-controlled
    gcsfuse --billing-project wb-glacial-potato-8710 --implicit-dirs vwb-aou-datasets-controlled ~/mnt/aou-controlled
    ```
    **`--implicit-dirs` is not optional** — without it nested paths silently don't resolve (`ls` looks empty) because GCS has no real directories and this bucket has no directory-placeholder objects. This gcsfuse (v3.10.0, preinstalled) daemonizes automatically — no second terminal. Then `~/mnt/aou-controlled/…/<file>.cram` works with plain `samtools view`. **When debugging remote/mount access, always redirect output to a file and check `$?` — never pipe through `tail`/`grep` while diagnosing; a filtered pipe once silently hid a real error behind stale-looking success output.** Public buckets (e.g. the Broad reference bucket) don't have this problem — plain `gs://` works for those.
12. **Pipeline outputs live under `~/pipeline_outputs/<person_id>/`, not `/tmp`.** `/tmp` isn't visible in the Jupyter file browser (confined to `$HOME`) and reads as disposable; `~/pipeline_outputs/` is the same persistent disk, just visible.
13. **The `has_lr_whole_genome_variant` BigQuery flag does not guarantee a usable aligned BAM — and after THREE rounds of path-pattern guessing each missing a real source, the fix is to stop guessing path patterns entirely and verify file existence directly.** `v9/wgs/long_read/manifest.tsv` has 15,424 rows / 14,521 unique people across 5 platform labels (revio 11,070, ont-r10.4.1 2,020, sequel2e 1,219, sequel2 991, ont-r9.4.1 124). This quirk's history is the actual lesson, not just its conclusion:
    - **Round 1:** filtered on `platform=='revio' AND '/revio/' in grch38_bam` — missed `sequel2`'s real BAM entirely (published under a different column, `grch38_haplotagged_bam`), and never considered that a platform-label filter could exclude a person's *other, resolving* manifest row (~879 of 14,521 people have more than one row).
    - **Round 2:** generalized to "any row whose BAM path is rooted under `v8_delta`, either column" — confirmed correct for `revio`, `ont`, and `sequel2e` (verified with `samtools quickcheck` + header + real HLA-window reads, across multiple centers), giving 1,772 people. But this was **still a guessed pattern**, and it missed `sequel2` *again*: `sequel2`'s real, already-confirmed-working BAM (991 people, verified earlier) apparently lives under yet another release folder that doesn't literally contain `/v8_delta/` — the "0 people resolved via `grch38_haplotagged_bam`" in round 2's own output was the tell.
    - **Round 3 (current, final): stop pattern-matching folder names altogether — verify each candidate file's existence directly** (`os.path.exists()` against the mount) for both known BAM-holding columns (`grch38_bam`, then `grch38_haplotagged_bam`), per row, then dedupe to any resolving row per person. This is immune to however many more release-folder-naming surprises AoU's manifest has, because it never depends on knowing the folder name — only on whether the file is actually there. Slower (a few minutes per cohort build, one `os.path.exists()` per candidate column per row, ~2×15,424 mount stats) but correct by construction, not by how many special cases have been enumerated so far.
    - `v9_delta`-rooted rows (checked for `pacbio`- and `ont`-labeled rows, at BCM/JHU/UW) consistently resolve to a de novo phased assembly + variant-call product with no aligned BAM published — a different data product for that release increment, not a missing file; existence-checking correctly excludes these without needing to know that in advance.
    - **~879 of 14,521 people have more than one manifest row** (different center/platform combos for the same person — matches AoU's own docs: "some samples have two rows... need `research_id`+`center`+`platform` to uniquely identify"). Critically, **a person's newest row can be a dead end while an older row for the exact same person resolves fine** — directly confirmed: person `1008366`'s `revio`-labeled row points at a nonexistent `v9_delta/pacbio` path, but their separate `ont-r10.4.1`-labeled row is a real, existing BAM. **Resolution happens per row, then dedupes to any resolving row per person** — filtering by platform label alone (even a correct-looking filter) silently drops people who have a working row under a *different* platform label.
    - **The lesson (Marc, 2026-07-11): this was never "the dataset is broken."** Three rounds in a row, each felt like "the real, general rule" right up until the next check disproved it. AoU restructuring how each release batch is published (assembly-based vs. alignment-based; inconsistent column names across platforms; multiple rows per person; unknown-to-us release-folder names) is a normal consequence of a growing, actively-developed dataset, not a defect — but it also means **any pattern inferred from a handful of confirmed cases is a hypothesis about this dataset, not a property of it**, and should be tested against ground truth (does the file actually exist?) rather than encoded as a rule and trusted at scale. Two additional, compounding mistakes along the way: an unverified arithmetic claim ("`1,219 − 760 = 459`, must be blank") that was never actually checked and turned out false, and a silent Python operator-precedence bug (`&` binds tighter than `==`, so `a & b == False & c` doesn't parse as intended) that produced garbage output which briefly looked like a real result. **Fix, now standard practice for this kind of question: verify by checking whether the file exists, not by matching a path or column-name pattern — and print/read actual values before trusting a computed boolean mask.**
    - **`build_experiment_d_cohort.py`'s `find_existing_bam()` implements this directly**: for each manifest row, checks both known BAM-holding columns (`grch38_bam`, then `grch38_haplotagged_bam`) and calls `os.path.exists()` against the mount — no path-pattern or folder-name assumption at all. Resolving rows are deduplicated per person afterward, keeping any one that resolves. Slower than the earlier pattern-match versions (~2×15,424 mount stats, a few minutes per cohort build) but correct regardless of how many more release-folder or column-naming surprises this manifest still has.
    - **Confirmed FLOOR as of 2026-07-11: 2,763 of 14,521 unique people** have at least one manifest row with a real, existing GRCh38 aligned BAM under one of the two known columns (`grch38_bam`, `grch38_haplotagged_bam`) — up from the original undercount of 1,031. `sequel2`'s real files turned out to live under a **fourth** release folder, `v7_base` (never seen before this check) — each of the first three "this is the general rule" moments was wrong, so treat this number the same way: a floor we've directly verified, not a proven ceiling. Also confirmed: `grch38_bam` and `grch38_haplotagged_bam` are genuinely different files for `sequel2` (raw aligned BAM vs. a `MARGIN_PHASED.PEPPER_SNP_MARGIN.haplotagged` derivative) — `find_existing_bam()` correctly prefers the raw `grch38_bam` (checked first). The ancestry jump this unlocked is concentrated almost entirely in AFR (386 → 1,326): `sequel2` is 940/991 (94.9%) AFR, AoU's deliberately African-American-enriched original long-read pilot cohort (`reference/AOU_DATA_ACCESS_NOTES.md`) — not a data artifact, and this alone is well beyond what a balanced ancestry-stratified design needs; MID (27) and SAS (77) remain the practical bottleneck for an equal-n-per-group cohort.
    - **Open, unresolved (2026-07-11): the remaining ~11,758 people are NOT proven to lack real long-read data — only unresolved via what we've checked so far.** AoU's own CDRv9 release notes (support site, June 2026) advertise "more than 14,000 long-read genomes" as a headline number, which is hard to square with only ~19% being sliceable today, and three consecutive rounds of "found the real rule" being wrong is a strong prior against treating this as settled. What we've actually verified: exhaustively checked only the 62 columns of *this one* manifest file (`v9/wgs/long_read/manifest.tsv`) for the 2 known BAM-holding columns, and confirmed (via full directory listings, not guesses) that specific `v9_delta`-rooted paths for `pacbio`/`ont` rows at BCM/JHU/UW genuinely have no bam-holding folder *at those exact locations*. We have NOT checked: whether a different or newer/supplementary manifest exists (the org PDF may not be exhaustive — worth checking the Workbench UI's Data Dictionary / Dataset Builder directly, and asking Aleix, who may already know); whether `revio`-platform `v9_delta` rows at centers other than BCM (e.g. JHU, UW) follow the same assembly-only pattern or differ; or whether AoU's "14,000+" figure itself refers to something broader (e.g., including assembly-only or otherwise-typed genomes) than "has an aligned GRCh38 BAM." Worth a dedicated follow-up before treating 2,763 as anything more than "what we can use today."
    - **Round 4 (2026-07-21): superseded — AoU backfilled the gap, this is no longer 2,763, it's all 14,521.** `scripts/lr_manifest_format_census.py`'s exhaustive per-column census (DECISIONS.md, `reports/lr_data_census/README.md`) found every one of the 14,521 people now has a real `grch38_bam`. Directly spot-checked against this quirk's own previously-documented dangling case — person 1008366's `revio` row, named above as pointing at a nonexistent path — and confirmed it now resolves as real. Not a bug in either investigation: the data itself changed in the ~10 days between the two checks. **Lesson reinforced, not contradicted: "verify by real existence, not by trusting a prior result at face value" cuts both ways — a carefully-established number can still go stale on an actively-developing dataset, and re-checking it (rather than assuming three rounds of correction meant it was finally permanent) is what caught this.**
14. **The VM auto-stops after ~1 hour with no processes running — not "1 hour of inactivity" in a vague sense.** Correction (Marc, 2026-07-21): a long-running foreground job keeps the VM up for its entire duration, however many hours that is — the idle clock only starts once the process exits. So a job finishing and *then* the VM stopping an hour later (wiping the terminal) is a materially different failure mode from the VM killing a job mid-run — see quirk #22's corrected account, which originally misdiagnosed this. Practical upshot is the same either way: **assume the VM (and its terminal sessions) has been restarted at the start of every session.** Consequences, and a large fraction of session-start errors trace to them:
    - **The gcsfuse mount does not survive a stop/restart** — it's gone every new session. Remount (quirk #11) *before* anything that reads bucket data. A start-of-session `FileNotFoundError` on `~/mnt/aou-controlled/…` almost always means "the VM slept, remount," not a real path problem.
    - **After remounting, the FUSE layer needs a beat to become ready.** gcsfuse prints `File system has been successfully mounted` but a command fired in the *same paste* can still race ahead and get `FileNotFoundError`. Run the mount on its own, then **verify with `ls` of a known path** (which also warms the implicit-dir metadata cache) before running dependent commands — don't chain the mount and the consumer in one paste (same lesson as quirk #2).
    - **Running processes and `/tmp` don't survive; the persistent home disk does.** `~/pipeline_outputs/`, `~/repos/`, `~/tools/`, `~/ref/` all survive the restart fine — only the mount, background processes, and the activated pixi shell need redoing.
16. **Check `reference/` before doing any external research (web fetch, search, browser) for AoU documentation.** The 59-page "How the All of Us Genomic Data are Organized v9.pdf" is already saved in `reference/`, and `reference/AOU_DATA_ACCESS_NOTES.md` already has a running log of what's been extracted from it (bucket paths, schemas, confidence markers). **Confirmed 2026-07-10:** re-derived the genetic-ancestry TSV's location and schema entirely via live web search + browser automation (support.researchallofus.org search, a blocked 403 fetch, then manually paging through the same PDF via Chrome), when `AOU_DATA_ACCESS_NOTES.md` section 6 already had the schema written down from this exact PDF, sourced in a prior session — the only missing piece (the literal bucket path) took real time and tokens to re-find that a `grep reference/AOU_DATA_ACCESS_NOTES.md` and a `Read` of the local PDF would have gotten in seconds. **Fix, now standard practice: `grep` (or read) `reference/AOU_DATA_ACCESS_NOTES.md` and check for a matching PDF/doc in `reference/` first — only go external if the local notes don't cover it.**
17. **SpecImmune must always run inside the `specimmune` pixi env — never assume the ambient shell's env doesn't matter.** Its long-read pipeline shells out to `sniffles` (SV caller), which is only on `PATH` inside `specimmune`, not `spechla` (which has `samtools` but not SpecImmune's own deps). Running `python3 scripts/main.py` from a `spechla`-activated shell fails deep inside the pipeline (`sniffles: command not found`) on every gene, cascading into missing-VCF errors — **but `main.py` still exits 0**, so a naive `$?` check reports success while producing zero real output. Confirmed 2026-07-09: an entire unattended 9-config sweep silently produced only NA rows this way. **Fix, now standard practice: invoke SpecImmune via `pixi run --manifest-path ~/repos/pilot-validation/pixi.toml -e specimmune -- python3 scripts/main.py ...`** instead of relying on `cd`-and-hope — this pins the environment regardless of what shell launched the script. **Also: after any automated SpecImmune call, verify the actual expected output file exists — never trust exit code alone** (`run_aligner_pad_sweep.sh` does both now).

18. **SpecHLA can silently return mostly-empty per-gene results while exiting 0 and writing a real `hla.result.txt`.** Confirmed 2026-07-10: a genuinely starved input (gene-cluster window truncated to 100-5000bp, well below what's needed) made SpecHLA finish in 22-62 seconds — 4-11x faster than the ~4m14s floor seen with adequate data — because it appears to skip/abort per-gene processing when a gene has too few reads, rather than erroring out. The result file still existed and passed the standard "does the expected output file exist" check used elsewhere in this project, even though most of its 8 genes were blank (`-`). **Fix: any automated SpecHLA sweep must also check per-gene completeness (count non-`-` genes in `hla.result.txt`, or its line count), not just file existence** — same category of lesson as quirk #17 (SpecImmune/sniffles: exit 0 ≠ actually worked), a different failure mode, same rule: a coarse success signal (exit code, file existence) is not sufficient on its own for either tool.

19. **The pipeline generates ~3 GB of intermediates per person (almost all from SpecImmune) — with no cleanup, the ~138 GB disk fills at ~person 10 and every later person fails instantly with `No space left on device`.** Confirmed 2026-07-11: an unattended Experiment D run completed 10 people, then person 11's SpecImmune wrote partial output as the disk hit 100%, and persons 12-60 all failed at pixi env activation (`/tmp` is on the same full overlay). The failures are clean/retryable (resumable design), but the run is dead until space is freed. **Disk policy, now enforced:** the only must-keep outputs are tiny (`comparison_log.csv`, tool result files `hla.result.txt`/`*.formatted.txt`, `*_timing.txt` — all <1 MB); everything bulky (sliced BAMs, FASTQs, tool alignment/assembly intermediates) is regenerable or redundant once the calls are in `comparison_log.csv`. Rule: **inside a person dir, anything >1 MB is disposable.** `run_experiment_d.sh` now auto-prunes each person the moment they're marked done (drops it from ~3 GB to a few KB, keeping disk flat across the whole cohort); `prune_pipeline_outputs.sh` applies the same policy on demand (`--dry-run` to preview) for cleaning old closed-experiment cruft. We never persist raw CRAM/BAM (streamed from the mount) and no longer persist the sliced versions either (re-slicing costs ~5 s).

20. **`bash`'s `>` redirect needs the target directory to already exist** — `cmd > ~/newdir/out.txt` fails with `No such file or directory` if `~/newdir` doesn't exist yet, even if the script itself would `os.makedirs()` it internally — bash resolves the redirect target before the command ever runs. Always `mkdir -p` the output directory first, as its own step, before any redirected command.
21. **IPD-IMGT/HLA's `Allelelist_history.txt` encodes each release version as a plain digit string, not `X.Y.Z`** — e.g. `3650` = release 3.65.0, `3380` = 3.38.0, and the single oldest column `116` = 1.16.0 (pre-dates the patch-number convention, patch assumed 0). Decode rule: last digit = patch, preceding 2 digits = minor, remainder = major. Confirmed correct by cross-checking against two independently-known versions already in this project (SpecHLA's DB 3.38.0, SpecImmune's DB 3.64.0 — both decoded correctly). See `experiment_a3_allele_space_coverage.py`'s `decode_version()`.
22. **Any script that only prints its real output (never writes it to disk incrementally) can lose that output to session/terminal loss even if the process itself completes successfully — this is a different, easier-to-hit failure mode than "the VM killed a running job."** Real incident, 2026-07-21, `lr_manifest_format_census.py`'s first version: it did all existence-checking (16 columns x ~15,424 rows) and only wrote a file on the very last lines of `main()`, printing all its actual distribution tables to stderr and nothing else. **Initial (wrong) diagnosis: assumed the VM restart killed the process mid-run at ~5h.** **Corrected by Marc:** per quirk #14, a busy VM doesn't stop mid-job — the real sequence was almost certainly: the ~5h run completed successfully and printed everything to the terminal; the VM then sat idle for an hour *after* it finished and stopped, wiping that terminal's scrollback with it (no tmux/screen or output redirection was in use, so nothing survived that could be re-attached to). Separately — and this is what actually destroyed the on-disk result — the script's `--sample-only` mode and its real full-run mode wrote to the *same default output path*, and a subsequent sanity-check invocation (`--sample-only 50`, run after the full result already existed, per this project's own multi-agent workflow — see DECISIONS.md/STATUS.md on concurrent editing) silently overwrote the real ~14,521-row result with 50 test rows. Both root causes are real and compound: printed-only output is invisible to anything but a live terminal, and a shared output path across a "real" and a "test" mode is a second, independent way to lose a result even if the terminal problem didn't exist. **Fix, now standard practice for any job expected to run past a few minutes, regardless of exactly which of these bites first: (a) write real output to disk incrementally, don't rely on a terminal staying open — process in chunks (a few hundred rows), append+fsync each chunk to a checkpoint file immediately, and make reruns detect + resume from the last completed chunk rather than restart from zero** (mirrors `run_experiment_d.sh`'s existing per-person resumability, just at finer granularity); **(b) a throwaway/test invocation of a script must never share an output path with its real-run mode** — `lr_manifest_format_census.py` now gives `--sample-only` its own separate path unconditionally. Also cut real runtime by parallelizing the checks (`ThreadPoolExecutor` — network-I/O-bound over FUSE, not CPU-bound, so threading helps despite the GIL) and by dropping companion/index files (`.bai`/`.tbi`/`.csi`/`.pbi`/`.gzi`) from the checked set, since they're near-certain to exist whenever their primary file does and were roughly doubling the check count for no new information.

## Package management

Deps for SpecHLA and SpecImmune are managed with [pixi](https://pixi.sh), not conda — two separate pixi envs (`spechla`, `specimmune`) in one `pixi.toml`, mirroring how upstream ships them as two never-designed-to-coexist conda envs. Deps are loosely pinned rather than a full port of upstream's exact-pin `environment.yml` (SpecHLA's own `CONDA_PACKAGING_PROPOSAL.md` calls its pins outdated/overly strict; SpecImmune's has 400+ auto-exported pins not worth hand-porting). SpecHLA itself isn't conda-packageable — it vendors pre-compiled binaries (`bin/bcftools`, `bin/novoalign`, `bin/fermikit/`) called by hardcoded relative path, so pixi supplies only the surrounding toolchain; the tool is cloned from source and built via `bash index.sh`.
