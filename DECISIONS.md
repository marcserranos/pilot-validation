## Open questions

### Strategic
- **AoU-native callset: trust & sequencing.** AoU already ships a full-cohort srWGS HLA callset (HLA-HD + Polysolver + OptiType) — this reframes the "massive calling is our contribution" pitch. Fork: **(a)** if AoU-native holds up vs. long-read → pivot toward validate-then-build-downstream on the existing calls (faster path to the aims); **(b)** if it shows ancestry-correlated discordance → the calling work becomes *more* necessary and the bias finding is itself novel (every existing AoU-derived HLA study would inherit that bias). **Lean: validate against long-read first** — an ancestry-distribution study on AoU's calls is circular if we don't yet know whether those calls are ancestry-biased in exactly the dimension being studied. Questions for supervisors: were they aware AoU ships this? any known ancestry benchmarks of that specific ensemble? validate-first vs. study-first preference? which outcome do they expect (shapes whether the pilot needs to be stratified/larger to detect ancestry bias vs. just overall concordance)? (raised 2026-07-09).
  - **n=3 evidence (2026-07-09, EXPERIMENTS.md):** AoU-native is broadly concordant for Class I / DQB1 / DPA1, **but is the lone outlier at DQA1 in 3/3 people** — and this one is cross-technology robust: a short-read caller (SpecHLA) *and* a long-read caller (SpecImmune) agree against it, so it isn't a short-read-shared-bias artifact. Exactly the locus-specific bias the "validate first" stance guards against — concrete early support for validating before building downstream on AoU's calls, at least locus-by-locus. Not yet ancestry-resolved.
  - **Mechanism decided 2026-07-09: Experiment D (EXPERIMENTS.md roadmap) is now how this gets answered at scale** — the ancestry-stratified 3-way comparison directly targets whether the DQA1 bias (and the DPA1 divergence) is ancestry-correlated or universal. **The supervisor conversation is no longer a blocking prerequisite for starting that pilot** (see Resolved below) — it stays open and valuable, just not gating.
  - **Ancestry-distribution evidence (2026-07-10, EXPERIMENTS.md Experiment A extension):** the two loci split cleanly. DPA1's homozygosity varies ~2.5x by ancestry (afr 27% → eur 68%), tracking the expected Out-of-Africa diversity gradient — looks like real population structure, not an AoU-native calling artifact. DQA1 shows no such gradient (flat ~18-19% across afr/amr/eas/eur/sas) — consistent with its discordance being method/database-specific rather than population-genetic. Doesn't resolve whether the actual SpecHLA/SpecImmune-vs-AoU-native divergences are ancestry-correlated — still Experiment D's job.
- **Can SpecImmune-LR serve as the validation reference on this data?** The pilot treats long-read as "presumed most-accurate," and that prior is reasonable for this region. At n=3 SpecImmune-LR diverges from the short-read consensus at 6/24 loci — but **concordance cannot tell us whether those are SpecImmune errors or short-read blind spots**, because the two agreeing methods (AoU + SpecHLA) share short-read data and so aren't independent corroboration. Divergences range from deeper resolution (expected, trust it) to a different allele *group* (A\*02 vs A\*24, warrants scrutiny). **To adjudicate: pull per-call confidence (`One_guess` vs ambiguous) + measure per-locus long-read depth from the sliced BAMs.** Only then can we say whether SpecImmune is a clean reference here, or whether its reliability is coverage/config-limited (8× HiFi on `grch38_noalt`, missing the ALT contigs its docs want). Do **not** dismiss SpecImmune divergences as "over-resolution" — resolving deeper is the point of using long reads. (raised 2026-07-09).
- **Data egress / compliance.** HLA genotypes are near-fingerprint-identifying; even small derived files may need AoU egress/publication review before leaving the Workbench, regardless of size. Not confirmed with the Controlled-Tier sponsor. Until then: do comparison/analysis *inside* the Workbench, don't download results.
- **Bare `person_id`s in a public repo.** Operational commands (STATUS, EXPERIMENTS) carry bare research_ids. Genotypes are quarantined to the gitignored `.local.md`, but whether bare ids belong in a *public* repo at all is unresolved — flag to the sponsor alongside egress. (Current practice preserved, not expanded; the ids are already in git history.)

### Methodological
- **minimap2 vs bwa for SpecImmune read-binning.** `--align_method_1 minimap2` = faster (gap shrinks as padding narrows: ~33min at fullpad → ~1.7min at pad5k) but caused a real DPB1 regression on person 1017156. **2026-07-09 update (person 2522883, full aligner x padding sweep, EXPERIMENTS.md):** DPB1 did NOT regress — both aligners gave identical calls/identity at every padding level, and no gene showed any accuracy difference between aligners in this person. At n=2, leans toward "one-off, not systematic," but **still not adopted as default — confirm on a 3rd person** before changing the standing choice.
- **Region padding — real data now available (2026-07-09, EXPERIMENTS.md).** pad100k (3.9x smaller than current 4Mb default) showed zero degradation across all 8 genes in person 2522883, and real runtime savings (~21min bwa vs ~50min). pad40k and pad5k showed measurable degradation on 2 of 8 genes (Gene A ambiguity explosion, DRB1 identity loss) — a real, not hypothetical, floor. **Candidate to adopt pad100k as new default padding for SpecImmune-LR — pending confirmation on ≥1 more person** (this is n=1 for the padding dimension).
- **SpecHLA-SR padding — separate finding, do not conflate with the LR result above (2026-07-10, EXPERIMENTS.md Experiment B).** Real fragment-size floor measured at 650bp (p99), ~50x smaller than the LR floor — as expected, since short reads are ~50x shorter than long reads. Zero call degradation observed at ANY of the 8 genes down to 325bp padding (below the measured floor), unlike the LR sweep which found real degradation at DRB1/Gene A. **Candidate to adopt pad2000-pad10000 as new default padding for SpecHLA-SR specifically — pending confirmation on ≥1 more person.** SpecHLA and SpecImmune now have two independently-measured, tool-specific padding recommendations — they should not be unified into one "padding" default.
- **4-field vs 2-field resolution** — is 4-field actually required downstream, or is 2-field enough for the frequency/PRS aims? **Partially resolved 2026-07-10 (EXPERIMENTS.md, Experiment A):** AoU-native is structurally capped at 3-field (99.4-100% of calls per locus, zero 4-field calls across 535,658 people) — if 4-field turns out to be required, AoU-native cannot supply it under any circumstances and SpecHLA/SpecImmune become load-bearing, not just a validation check. Still open: whether the aims actually need 4-field.
- **Reference panel choice** and its effect on miscalls in non-European ancestries.
- **`grch38_noalt` compatibility.** lrWGS BAMs are aligned to `grch38_noalt` (no ALT/HLA contigs); SpecImmune's docs want ALT contigs for divergent-allele sensitivity. No workaround (AoU ships no ALT-contig lrWGS). Watch long-read call completeness/confidence vs. short-read in the bake-off.
- **Ancestry variable for the scale-up.** Prefer AoU's genetic-ancestry TSV (PCA-based AFR/AMR/EAS/EUR/MID/SAS/OTH — 7 categories, keyed by research_id) over the self-reported race field the original cohort query used, for the stratified n=25/100 pick.
- **Compute backend for scaling** — Workbench-native likely; GPU is a mismatch (HLA calling is CPU/IO-bound). Needs a per-method cost model: AoU-native = $0 egress (pre-computed variant tier); SpecHLA/SpecImmune both pull raw CRAM/BAM = egress cost.
- **Genomic Quality Report (v9)** — has GiaB-based QC/coverage/sensitivity methodology relevant to the coverage/error-profile aim. Only archived v6–v8 versions found so far; look for v9 in the Workbench UI.

## Resolved decisions

- **Fuse "n=25/100 scaling" with the ancestry-stratified 3-way comparison (2026-07-09).** Marc: these were being treated as two separate items (a horizontal/vertical-scaling question, and a separate n=5-10-per-ancestry proposal) but they're the same experiment. The scaling-approach decision (parallel VMs vs. single-VM optimization) now folds into that one experiment's execution plan (Experiment D, EXPERIMENTS.md) rather than being planned separately.
- **Proceed to the ancestry-pilot without waiting for the supervisor conversation (2026-07-09).** Reason: compute cost is low enough (~$10, well under 40 hours serial even fully unoptimized) that gating a cheap pilot behind a conversation that can happen in parallel isn't worth the delay. The "AoU-native trust" supervisor conversation stays open and valuable — it's just no longer a blocking prerequisite for starting Experiment D.
- **SpecImmune gene-panel restriction reconsidered (2026-07-09).** Previously ruled out (2026-07-08) as not worth speculative effort. Revisited given real padding payoff evidence — a real chunk of SpecImmune's runtime is fixed per-gene overhead scaling with the ~30-gene panel, not input size, so restricting to the 8 classical genes could be a bigger lever than padding alone. Queued as Experiment C (EXPERIMENTS.md) — investigate `make_db.py` directly before assuming it's buildable.
- **SpecImmune gene-panel restriction: confirmed feasible, 2026-07-08 "ruled out" call overturned (2026-07-10).** Read `make_db.py`/`main.py`'s actual source (local clone, not the VM). The `--HLA_fa`/`--HLA_exon_fa` mechanism genuinely restricts the DB, and every per-gene typing stage discovers genes dynamically from the `--db` folder's contents (`get_folder_list()`), not a hardcoded list — restriction really works. The 2026-07-08 "not worth speculative effort" call was itself caused by an actual bug (`UnboundLocalError` in the `--HLA_fa` branch) that made the lever appear broken; nobody had gotten past it to see the mechanism worked. Bug found, fixed, and packaged (`patch_specimmune_for_gene_restriction.py`). Local build-time evidence: restricted DB ~107s vs full ~225s (~2.1x). Full local accuracy comparison hit an unrelated macOS/samtools race condition (see EXPERIMENTS.md) — finishing the accuracy comparison on the VM is the next step, not further local debugging.
- **Ancestry-cohort availability confirmed by Marc (2026-07-09).** The concern that AoU's long-read sub-cohort might be too ancestry-skewed to support n=5-10 per group (raised 2026-07-09, based on historical cohort composition notes) is resolved: an existing matched SR+LR cohort (likely `HLA_Pilot_Matched_SR_LR_100`, 9,358 candidates) already covers adequate people across ancestries. No separate feasibility check needed before Experiment D — still apply the `/revio/` BAM filter (quirk #13) per candidate when actually picking individuals, independent of which cohort resource is used.
- **3-way bake-off, not a 2×2 (2026-07-08).** SpecImmune can't do short-read (argparse `-y` only accepts `nanopore`/`pacbio`/`pacbio-hifi`); AoU has no native long-read caller. So the real comparison is AoU-native-SR vs. SpecHLA-SR vs. SpecImmune-LR — two of the four naive cells don't exist.
- **Skip SpecHLA's long-read mode (2026-07-08).** SpecHLA's own README defers to SpecImmune for long-read — no reason to test a mode its authors don't recommend.
- **"Presumed most-accurate reference," not "ground truth," for SpecImmune-LR (2026-07-08).** No independently-validated clinical typing set exists in this pilot; long reads are presumed better for this hyper-polymorphic region, not proven-correct in an absolute sense. Keep the distinction in any write-up.
- **Smoke test = 2–3 people (2026-07-08).** Not increased despite AoU-native being free — SpecHLA/SpecImmune compute + debugging time is the bottleneck, not sample count.
- **Candidate selection filters to `/revio/` BAM paths (2026-07-08).** The eligibility flag doesn't guarantee a real aligned BAM (ENVIRONMENT quirk #13). Build the filter in from the start.
- **gcsfuse mount, not raw `gs://` (2026-07-08).** htslib doesn't cleanly bill the requester-pays bucket. Recipe in ENVIRONMENT quirk #11.
- **Disable SpecImmune visualization, `--visualization ""` (2026-07-07).** SVG→PDF converter missing, no sudo; visualization is cosmetic, irrelevant to allele calls. Empty string is the only value argparse's buggy `type=bool` treats as false.
- **Pipeline outputs to `~/pipeline_outputs/`, not `/tmp` (2026-07-08).** `/tmp` isn't visible in the Jupyter file browser (quirk #12).
- **Dropped the 1000 Genomes pilot.** Validating directly on AoU's own SR/LR overlap population instead.
- **pixi, not conda** — see ENVIRONMENT / package management for the full rationale.
